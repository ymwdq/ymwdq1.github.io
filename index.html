前言

位图法就是bitmap的缩写。所谓bitmap，就是用每一位来存放某种状态，适用于大规模数据，但数据状态又不是很多的情况。通常是用来判断某个数据存不存在的。

数据结构

unsigned int bit[N];
在这个数组里面，可以存储 N sizeof(int) 8个数据，但是最大的数只能是N sizeof(int) 8 - 1。假如，我们要存储的数据范围为0-15，则我们只需要使得N=1，这样就可以把数据存进去。如下图：

假如数据为[5，1，7，15，0，4，6，10]，则存入这个结构中的情况为：

使用场景

  • 大量数字排序。

给 10W 个不重复的整数，这些数的范围在 0 到 100W 之间，如果我们直接对这些数字使用快排，时间复杂度大约为 o(logn)，如果我们使用位图那么时间复杂度为 o(1)。

  • 大量数字去重。

给 2.5 亿个整数，找出其中不重复的数字。

采用2-Bitmap（每个数分配2bit，00表示不存在，01表示出现一次，10表示多次，11无意义）。其实，这里可以使用两个普 通的Bitmap，即第一个Bitmap存储的是整数是否出现，如果再次出现，则在第二个Bitmap中设置即可。这样的话，就可以使用简单的1- Bitmap了。

位图法的限制

  • 存储空间分配依赖于数字的范围

如果数字特别大，那么就有可能无法分配足够的内存。

  • 可读性差

海量数据处理问题(3) - 字典树

发表于 2018-03-14 | 分类于 算法 ， 大数据处理

前言

我们在使用google等搜索引擎或者淘宝、京东等购物网站的时候，经常会搜索框，我们在搜索框输入一个词汇之后，搜索框总是可以向下扩展一个列表，列出所有选项，你有想过这个功能是怎么实现的呢？

没错，答案就是字典树，也叫 trie 树（当然现代的搜索引擎，引入了机器学习算法对结果进行优化，这部分就属于推荐系统的范畴了）。

trie 树的定义

这里直接给出维基百科的定义

In computer science, a trie, also called digital tree and sometimes radix tree or prefix tree (as they can be searched by prefixes), is a kind of search tree—an ordered tree data structure that is used to store a dynamic set or associative array where the keys are usually strings.

trie 一般长成这样

简单来说，trie 树就是一种特殊的多叉树，里面存储的是字母等自然语言常见的元素。如果我们从树的根节点走到叶子节点，就可以组成一个完整的单词。

设计初衷

有人可能会问，为什么查找单词、词频统计可以用 trie 树，他有什么优势？试想，现有一堆单词，你需要查找这堆单词里是否存在 yangming 这个单词你会怎么做？首先肯定要找以 y 开头的单词，找到 y 开头的单词之后，我们再找第二个字母为 a 的单词，如此循环往复。

这时，一个树的结构出现在脑海中！

trie 树的典型应用场景

  1. 前缀匹配。即使是在海量数据的情况下，亦可以做到 o(logn) 级别的匹配，如果使用朴素的遍历方法也要o(mn) 的时间复杂度。
  2. 词频统计，尤其是大数据的情况下。可能有人要说了，词频统计简单啊，一个hash或者一个堆就可以打完收工，但问题来了，大数据情况下，内存有限呢，还能这么玩吗？所以这里我们就可以用trie树来压缩下空间，因为公共前缀都是用一个节点保存的。
  3. 词汇查找。优势和上面一样，时间复杂度低，空间利用率高。

trie 树的实现

其实看到定义之后可以发现，trie 树的定义不像红黑树、B 树、AVL 树那样复杂，插入也是。下面直接给出字典树的 Python 简单实现。

总结

我觉得其实 trie 树也是一种特殊的 HashMap，相比传统的基于 Hash 的散列表，我觉得它有以下几点优势

  • 空间利用率高。
  • 不用处理冲突问题
  • 可以接受的查找性能

海量数据处理问题（2）- Hash + 分治

发表于 2018-03-13 | 分类于 算法 ， 大数据处理

前言

本文将介绍Hash + 分治策略处理海量数据中的使用

引例

1. 求访问量最大的IP地址

给定一个大文件，文件存了海量IP，每个IP用换行符隔开，求出出现次数最多的IP。

1.1 求解步骤

  • Hash取模。顺序读取大文件的每一行，求对每一个IP求Hash后模1000的值，即 a = Hash(IP) % 1000，这样我们可以得到1000个不同的值，a0 到 a999，将相同a值的数据写到对应的文件中（f0,f1….f999）。这一步的作用是把相同的URL写入到相同的文件中，方便排序。
  • 顺序读取每一个小文件，对每一个文件使用一个HashMap统计各个 IP 出现的次数。
  • 求出每一个小文件 IP 出现次数最多的那个 IP
  • 最后汇总小文件统计出的最大次数 IP（一共1000个），求出次数最大的那个 IP 即为所求。

1.2 解析

  • 第一个步骤，我们相当于把数据做了一个归类，把相同属性的数据汇总到一个文件，方便进行计数和排序。
  • 第二个步骤，我们使用了HashMap 统计各个数据出现的次数
  • 第三个步骤，使用排序方法求 TopK，这里我们求的是 Top1，遍历一次即可
  • 第四个步骤，从 m 个小文件的 TopK 求出总体的 TopK

2. 海量查询记录求 Top 10

给定一个 1000W 查询记录（去重后有300W条记录），每条记录占 1 到 256 个字节，统计 Top10 查询记录，限定内存 1G。

求解步骤

  • HashMap 统计。假设每条记录占256字节，那么 300W 条记录大约占 256B * 3000000 ≈ 0.8GB，所以基本可以直接在内存里使用 HashMap 进行统计。
  • 堆排序，使用堆排序找出最大的 10 条记录。

3. 统计出现频率最高的 100 个词

给定一个 1GB 的词汇文件，统计出出现频率最高的 100 个词，限定内存 1MB。

这道题的难点主要在于内存的限制，内存一旦限制住，就很难继续进行排序，计数等操作。因此这时可以采用散列 + 分治的策略。

求解步骤

  • 散列。顺序读取文件，把每一个词汇散列到5000个小文件中，散列公式：Hash(query) % 5000。如果散列公式足够均匀，那么平均每个文件大小 200KB。足够我们放入内存进行计数、排序等操作。
  • 计数。使用 HashMap 或者字典树对每一个小文件的词汇进行次数统计。
  • 对每一个小文件，使用堆排序或者快排统计每个小文件的词频 Top100，并存入内存。
  • 再维护一个 100 大小的小顶堆，依次遍历这 5000 * 100 个数并更新小顶堆，最后求出 Top100。

举一反三

求相同的URL

给定两个8G的文件，均包含了若干个 URL，用换行符隔开，那么请问如何求同时存在于这两个文件当中的 URL？

总结

散列 + 分治的思想是我们处理内存、数据分散、最终排序的银弹。
通过散列，我们可以把分散的具有相同属性的数据汇总到一起（比如上题相同的查询词、相同的 IP 等等），汇总到一起之后方便我们进行计数、排序等操作。
通过分治，我们求出各个小部分的 TopK 最终通过堆排序等排序算法求出总体的 Topk，也可以通过对小文件进行排序，最后通过归并排序的方法，得到总体的排序。

  • 时间复杂度，这个很好理解，面对海量数据，时间性能差的算法缺点一定会暴露出来。
  • 空间复杂度，原因同上。

常见的问题

  • 给定一个大文件求数据项出现次数TopK
  • 给定若干个文件求数据项出现次数TopK
  • 给定一个大文件对文件中的数据进行排序

趣味算法 - LeetCode Solution.292 Nim Game

发表于 2018-03-07 | 分类于 算法

背景

前几天刷 LeetCode 遇到了一个有趣的小题，题目如下

You are playing the following Nim Game with your friend: There is a heap of stones on the table, each time one of you take turns to remove 1 to 3 stones. The one who removes the last stone will be the winner. You will take the first turn to remove the stones.
Both of you are very clever and have optimal strategies for the game. Write a function to determine whether you can win the game given the number of stones in the heap.

简单来说，就是有一堆石头，每次必须往这堆石头里取1到3个石头，规定谁最后取完石头谁就赢。现在假设你先取石头，并且你的对手会尽可能采取对自己有利的方式取石头，问给定 n 个石头，输出你是否能获胜。

比如如果给你4个石头，那么你是必定输的。

思路

先来看以下情况：

  1. 1 <= n <= 3，你是必胜的，一次取完就好。
  2. n = 4，你是必输的，因为你无论取多少，都会剩下1到3个石头.
  3. 那么 n = 5 呢， 你是必胜的，为啥？我们知道 n = 4 时，谁先取石头谁输，你可以的“陷害”你的对手，即取一个石头，那么到你的对手那一轮的时候，n = 4, 这时他就输了。
  4. 同理 n = 6 时, 你可以取走 2 个石头“陷害”的你的对手，迫使其失败。

总结规律，可以发现“陷害”策略，即当有n个石头的时候，采取“迫使”对手失败的策略，我们之关心 n - 1、n - 2、n - 3时是否必赢或者必输。

到这里我们很自然的就可以想到用动态规划的思想解题。递推式是

  • dp[n] = false (false 表示必输，true 表必胜)，当dp[n - 1]、dp[n - 2]、dp[n - 3] 都为 true 的时候。
  • dp[n] = true，当 dp[n - 1]、dp[n - 2]、dp[n - 3]有一个为 false 的时候，那么我们就可以采用“陷害”策略，嘿嘿。

采用“陷害”策略，用数组进行动态规划，时间复杂度 o(n)，空间复杂度 o(n)

也就是说，我们可以为任何一个 InputStream 的实现类提供带缓冲的 read() 等方法，很棒！这就体现了装饰器模式在不改动源码的情况下动态添加功能的思想。

其他具体装饰器

当然实现了 FilteredInputStream 的类有很多，他们都和 BufferedInputStream 类似，提供各种动态添加功能的方法，比如DataInputStream、PushbackInputStream。

使用装饰器模式的优缺点

  • 优点

    装饰器模式和继承都能拓展对象的功能，但是装饰器模式有着更大的灵活性。通过使用不同的具体装饰类以及这些装饰类的排列组合，可以创造出很多不同行为的组合。可以使用多个具体装饰类来装饰同一对象，得到功能更为强大的对象。

  • 缺点

    这种比继承更加灵活机动的特性，也同时意味着装饰模式比继承更加易于出错，排错也很困难，对于多次装饰的对象，调试时寻找错误可能需要逐级排查，较为烦琐。
