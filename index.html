记一次系统瓶颈排查与优化

发表于 2018-06-27 | 分类于 性能优化 ， 故障排查

一、背景

事情源于一个十分坑爹的 BUG，实验室下载了百万计的 APK 并解析存入数据库，存入数据库的其中一项就是 APK 的 MD5 值。

这没啥大不了的，直到有一天突然发现，前人写的 APK 解析工具类在算 MD5 值的时候少取了 1 位，本来存入数据的应该是一个 128 字节的字符串，但是实际存入的是一个 127 字节的字符串。

所以…实验室博士们经过商讨之后决定，对 200TB 的 APK 进行重新解析，这酸爽…

二、起因

重新解析过程很简单

  1. 从磁盘阵列读数据
  2. 读完一个 APK 后，解析
  3. 然后重新 UPDATE 数据库

这个程序是师弟写的。写的时候搞了几十个测试 APK 放到本地，然后写程序。写出来之后发现一个问题，就是很慢，一分钟最多处理 10 个左右 APK，不符合常理。师弟也尝试了多种办法。

师弟的第一个方法：多线程

在本地用了一个固定的 10 个线程的线程池，尝试用多线程处理，结果还是一分钟处理 10 个左右。照这个速度下去一年都跑不完。师弟百思不得其解，于是师弟有了第二个想法。

师弟的第二个方法：把程序放到服务器跑

实验室的服务器配置极高，内存有 200+G、CPU 核心数 64 核，4 路 GTX 1080Ti。平时一般用来跑深度学习等大型任务。放到服务器跑了之后，结果更奇葩…上了服务器处理速度不仅没有提高，反而变慢了，变成了一分钟处理 8 个左右…

师弟百思不得其解，最后找到了我。

三、瓶颈定位

师弟找到我之后，我首先怀疑是不是他的多线程代码部分是不是有问题，所以首先看了一下他写的代码，发现多线程写的没啥大毛病。一开始我也疑惑了一下，但是事实摆在面前：

  1. 程序处理速度异常慢，多线程不解决问题，肯定某个地方出问题了
  2. 放到服务器上跑，速度不增反降。

思考一下后，我实现了我的解决方案

使用 htop、iostat 命令初步观察

遇到这种性能问题，我认为应该首先定位故障、瓶颈出在哪儿，而不是一味的祭出多线程、对配置之类的想法。所以我让师弟在运行程序的时候用 htop、iostat 命令，观察一下程序资源占用状况。

观察到的结果是：CPU 占用率不到 150 %，磁盘占用率比较高。这时初步有了推断：程序的瓶颈不在 CPU，而在于磁盘 IO，这就解释了为什么多线程起不了作用。

编写测试类验证想法

为了验证我的想法，我再师弟写的程序里搞了几个计时器，对多文件操作、解析操作和写回数据库操作做了计时。结果如下

  • 读文件操作： 几百毫秒
  • 解析操作：几毫秒
  • 写回数据库：几毫秒

果然，多线程不起作用的原因找到了。程序瓶颈在磁盘 IO 那里，不管你开多少个线程磁盘的读取速度总是有限的，这就导致程序的处理速度和 IO 的性能成正比，因此要优化程序应该从提高磁盘 IO 入手而不是一味使用多线程。

为什么上了服务器跑反而变慢了

但是还有一个问题没有解决，为啥相同的程序放到了服务器跑变得更慢了。我仔细看了师弟的代码，发现在服务器上跑的时候，多了一步从分布式文件系统下载 APK 文件的操作。

本地跑的流程为：读取APK -> 算哈希 -> 写回数据库（因为师弟已经事先下了几十个测试 APK，所以不用下载）
服务器上跑的流程为：从分布式文件系统下载 -> 读取APK -> 算哈希 -> 写回数据库

这就导致从服务器上跑速度反而更慢。

真正的瓶颈定位：交换机 + RJ45 网线

理论上跑的流程为：从分布式文件系统下载 -> 读取APK -> 算哈希 -> 写回数据库。这几个步骤里面，最耗时的是哪个呢？

我心中的答案是：下载，其次是从磁盘读取。因为一款普通机械硬盘的读取速度通常可以达到 几百兆每秒，而下载的时候，如果下载走普通网线，其速度极限是 1000Gb/s 约等于 125MB/s。

为了验证想法。我特地跑到了机房，观察服务器与磁盘阵列连接方式。观察到如下情况

  • 实验室磁盘阵列使用的硬盘架通过 RJ45 网线出来。
  • RJ45 网线在连到交换机上
  • 交换机再通过 RJ45 网线连接到服务器

于是就验证了我的想法，RJ45 网线的极限才 1000Gb/s 约等于 125MB/s，所以不管怎么堆服务器配置、搞多线程都是没用的。因为元数据获取的速度极限就是 125MB/s，假如一个 APK 平均大小为 10MB，那么每秒最多能够处理的极限就是：12 个。

四、问题的解决

原因找到了，磁盘阵列通过 RJ45 的网线加上一个不给力的交换机连接服务器，导致生产者永远无法满足消费者。

当然解决办法也很明了，更换给力的网线和交换机。磁盘阵列不用换，我查看了磁盘阵列的说明书，发现其支持光纤出口。经过调研，给博士建议买光纤 + 光纤网卡，服务器插上光纤网卡后通过光纤直连磁盘阵列，不用走 RJ45网线 + 交换机。

博士采纳了我的建议，几个星期之后，实验室来了几个师傅安装光纤以及光纤网卡，第一次见到光纤网卡，心理有些小激动。一共装了 2 个光纤网卡，这些网卡价格不菲一共要 1.3W RMB。下面是安装光纤的实况。

五、效果

最普通光纤的速度为万兆每秒，远远超过磁盘 IO 速度。安装完光纤的后发现，之前每秒只能处理 7 - 8 个APK，改为光纤直连 + 多线程之后每秒能够处理 100 个 APK，速度提升了 10 多倍。之前预估 1 年左右才能跑完数据，现在预估只要 1 个月左右就能搞定。

六、总结

通过这次优化让我学到了很多，也让我有了几点感悟

  • 在处理故障、优化性能的时候，优先确定瓶颈，而不是无脑堆线程、堆配置、改架构
  • 应当熟悉一些常见硬件的基本参数：比如 RJ45 网线的速度、磁盘速度之类的
  • 有时候瓶颈可能不光发生在软件层面，也有可能在硬件层面

使用场景

  • 大量数字去重。

海量数据处理问题(3) - 字典树

发表于 2018-03-14 | 分类于 算法 ， 大数据处理

前言

我们在使用google等搜索引擎或者淘宝、京东等购物网站的时候，经常会搜索框，我们在搜索框输入一个词汇之后，搜索框总是可以向下扩展一个列表，列出所有选项，你有想过这个功能是怎么实现的呢？

没错，答案就是字典树，也叫 trie 树（当然现代的搜索引擎，引入了机器学习算法对结果进行优化，这部分就属于推荐系统的范畴了）。

trie 树的定义

这里直接给出维基百科的定义

In computer science, a trie, also called digital tree and sometimes radix tree or prefix tree (as they can be searched by prefixes), is a kind of search tree—an ordered tree data structure that is used to store a dynamic set or associative array where the keys are usually strings.

trie 一般长成这样

简单来说，trie 树就是一种特殊的多叉树，里面存储的是字母等自然语言常见的元素。如果我们从树的根节点走到叶子节点，就可以组成一个完整的单词。

设计初衷

有人可能会问，为什么查找单词、词频统计可以用 trie 树，他有什么优势？试想，现有一堆单词，你需要查找这堆单词里是否存在 yangming 这个单词你会怎么做？首先肯定要找以 y 开头的单词，找到 y 开头的单词之后，我们再找第二个字母为 a 的单词，如此循环往复。

这时，一个树的结构出现在脑海中！

trie 树的典型应用场景

  1. 前缀匹配。即使是在海量数据的情况下，亦可以做到 o(logn) 级别的匹配，如果使用朴素的遍历方法也要o(mn) 的时间复杂度。
  2. 词频统计，尤其是大数据的情况下。可能有人要说了，词频统计简单啊，一个hash或者一个堆就可以打完收工，但问题来了，大数据情况下，内存有限呢，还能这么玩吗？所以这里我们就可以用trie树来压缩下空间，因为公共前缀都是用一个节点保存的。
  3. 词汇查找。优势和上面一样，时间复杂度低，空间利用率高。

trie 树的实现

总结

我觉得其实 trie 树也是一种特殊的 HashMap，相比传统的基于 Hash 的散列表，我觉得它有以下几点优势

  • 空间利用率高。
  • 不用处理冲突问题
  • 可以接受的查找性能

海量数据处理问题（2）- Hash + 分治

发表于 2018-03-13 | 分类于 算法 ， 大数据处理

前言

本文将介绍Hash + 分治策略处理海量数据中的使用

引例

1. 求访问量最大的IP地址

给定一个大文件，文件存了海量IP，每个IP用换行符隔开，求出出现次数最多的IP。

1.1 求解步骤

  • Hash取模。顺序读取大文件的每一行，求对每一个IP求Hash后模1000的值，即 a = Hash(IP) % 1000，这样我们可以得到1000个不同的值，a0 到 a999，将相同a值的数据写到对应的文件中（f0,f1….f999）。这一步的作用是把相同的URL写入到相同的文件中，方便排序。
  • 顺序读取每一个小文件，对每一个文件使用一个HashMap统计各个 IP 出现的次数。
  • 求出每一个小文件 IP 出现次数最多的那个 IP
  • 最后汇总小文件统计出的最大次数 IP（一共1000个），求出次数最大的那个 IP 即为所求。

1.2 解析

  • 第一个步骤，我们相当于把数据做了一个归类，把相同属性的数据汇总到一个文件，方便进行计数和排序。
  • 第二个步骤，我们使用了HashMap 统计各个数据出现的次数
  • 第三个步骤，使用排序方法求 TopK，这里我们求的是 Top1，遍历一次即可
  • 第四个步骤，从 m 个小文件的 TopK 求出总体的 TopK

2. 海量查询记录求 Top 10

给定一个 1000W 查询记录（去重后有300W条记录），每条记录占 1 到 256 个字节，统计 Top10 查询记录，限定内存 1G。

求解步骤

  • HashMap 统计。假设每条记录占256字节，那么 300W 条记录大约占 256B * 3000000 ≈ 0.8GB，所以基本可以直接在内存里使用 HashMap 进行统计。
  • 堆排序，使用堆排序找出最大的 10 条记录。

3. 统计出现频率最高的 100 个词

给定一个 1GB 的词汇文件，统计出出现频率最高的 100 个词，限定内存 1MB。

这道题的难点主要在于内存的限制，内存一旦限制住，就很难继续进行排序，计数等操作。因此这时可以采用散列 + 分治的策略。

求解步骤

  • 散列。顺序读取文件，把每一个词汇散列到5000个小文件中，散列公式：Hash(query) % 5000。如果散列公式足够均匀，那么平均每个文件大小 200KB。足够我们放入内存进行计数、排序等操作。
  • 计数。使用 HashMap 或者字典树对每一个小文件的词汇进行次数统计。
  • 对每一个小文件，使用堆排序或者快排统计每个小文件的词频 Top100，并存入内存。
  • 再维护一个 100 大小的小顶堆，依次遍历这 5000 * 100 个数并更新小顶堆，最后求出 Top100。

举一反三

求相同的URL

给定两个8G的文件，均包含了若干个 URL，用换行符隔开，那么请问如何求同时存在于这两个文件当中的 URL？

总结

散列 + 分治的思想是我们处理内存、数据分散、最终排序的银弹。
通过散列，我们可以把分散的具有相同属性的数据汇总到一起（比如上题相同的查询词、相同的 IP 等等），汇总到一起之后方便我们进行计数、排序等操作。
通过分治，我们求出各个小部分的 TopK 最终通过堆排序等排序算法求出总体的 Topk，也可以通过对小文件进行排序，最后通过归并排序的方法，得到总体的排序。

海量数据处理问题（1） - 挑战与策略

发表于 2018-03-13 | 分类于 算法 ， 大数据处理

背景

在对海量数据进行处理的时候，我们会遇到平时我们不会关注问题，比如排序：如果是对100个数进行排序，我们可以写出各种花式的排序方法，那如果是对2.5亿个数进行排序呢，你会遇到内存不足、时间过长等各种问题；再比如求TopK，从1000个单词里统计出现次数最多的单词，我们同样有多种解法，但如果是统计1000万个呢？

因此，对海量数据进行处理的时候，我们必须采取合适的策略、数据结构、算法才能达到理想的结果。

处理海量数据面临的挑战

  • 内存不足，数据往往一次性读入内存进行排序等操作。
  • 数据分散，数据可能分散在各个小文件当中，比如有一亿个URL，均匀分布在5个文件当中，求出现次数最多的URL。
  • 时间复杂度，这个很好理解，面对海量数据，时间性能差的算法缺点一定会暴露出来。
  • 空间复杂度，原因同上。

常见的问题

  • 给定一个大文件求数据项出现次数TopK
  • 给定若干个文件求数据项出现次数TopK
  • 给定一个大文件对文件中的数据进行排序
  • 给定若干个大文件，求出若干个大文件的公共元素

You are playing the following Nim Game with your friend: There is a heap of stones on the table, each time one of you take turns to remove 1 to 3 stones. The one who removes the last stone will be the winner. You will take the first turn to remove the stones.
Both of you are very clever and have optimal strategies for the game. Write a function to determine whether you can win the game given the number of stones in the heap.

简单来说，就是有一堆石头，每次必须往这堆石头里取1到3个石头，规定谁最后取完石头谁就赢。现在假设你先取石头，并且你的对手会尽可能采取对自己有利的方式取石头，问给定 n 个石头，输出你是否能获胜。

比如如果给你4个石头，那么你是必定输的。

思路

先来看以下情况：

  1. 1 <= n <= 3，你是必胜的，一次取完就好。
  2. n = 4，你是必输的，因为你无论取多少，都会剩下1到3个石头.
  3. 那么 n = 5 呢， 你是必胜的，为啥？我们知道 n = 4 时，谁先取石头谁输，你可以的“陷害”你的对手，即取一个石头，那么到你的对手那一轮的时候，n = 4, 这时他就输了。
  4. 同理 n = 6 时, 你可以取走 2 个石头“陷害”的你的对手，迫使其失败。

总结规律，可以发现“陷害”策略，即当有n个石头的时候，采取“迫使”对手失败的策略，我们之关心 n - 1、n - 2、n - 3时是否必赢或者必输。

到这里我们很自然的就可以想到用动态规划的思想解题。递推式是

  • dp[n] = false (false 表示必输，true 表必胜)，当dp[n - 1]、dp[n - 2]、dp[n - 3] 都为 true 的时候。
  • dp[n] = true，当 dp[n - 1]、dp[n - 2]、dp[n - 3]有一个为 false 的时候，那么我们就可以采用“陷害”策略，嘿嘿。

解法一

采用“陷害”策略，用数组进行动态规划，时间复杂度 o(n)，空间复杂度 o(n)

解法二

上述代码其实还可以优化，仔细观察可以发现，我们进行判断的时候，只依赖于前 3 步，即dp[n - 1]、dp[n - 2] 和 dp[n - 3]。

我们可以直接用三个变量保存前3步的结果，然后动态更新即可。这样空间复杂度可以降到 o(1)

总结

进过一系列优化，我们成功的把时间、空间复杂度从 o(n) 降到了 o(1)，通过这道题一共有两点感悟

  • 动态规划的优化问题，在推出递推式之后，我们可以考虑使用若干个变量对之前的状态进行保存，而不是数组。此题我们只用了3个变量保存之前的状态。其实我们在求斐波那契数列的时候也可以用到此法。
  • 要善于观察规律。我们往往可以套用一些经典算法或者思想去解决问题，但是这并不意味这这些算法、思想一定是最优的，要具体问题具体分析。

定义

哈希技术也叫信息摘要技术，这里直接给出维基百科的定义

A cryptographic hash function is a special class of hash function that has certain properties which make it suitable for use in cryptography. It is a mathematical algorithm that maps data of arbitrary size to a bit string of a fixed size (a hash) and is designed to be a one-way function, that is, a function which is infeasible to invert.

简单来说，密码学中 hash 函数用于把一堆数据映射成若干个比特，并且无法恢复，也就是说 hash 函数式不可逆的。如下图所示。

性质

  • 确定性。相同的信息，只能产相同的信息摘要。

  • 不可逆。除非采用 Hash 碰撞技术
